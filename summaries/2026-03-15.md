### My fireside chat about agentic engineering at the Pragmatic Summit - (Simon Willison's Weblog)

https://simonwillison.net/2026/Mar/14/pragmatic-summit/#atom-everything

# Summary

Simon Willison discusses agentic engineering adoption, describing how developers progress from using ChatGPT for questions to trusting AI agents like Claude Opus 4.5 to write entire code modules. He emphasizes that while reviewing every line of code generated by AI is impractical, developers should treat AI-generated code similar to how they trust professional teams' services—verifying functionality through tests rather than manual code review.

Willison highlights key practices for effective AI-assisted development: using test-driven development (writing tests first), manual testing with tools like his Showboat utility, and maintaining code quality standards that agents will naturally replicate. He also addresses critical security concerns, particularly the "lethal trifecta" vulnerability where agents with access to private data, exposure to malicious instructions, and exfiltration capabilities pose serious risks. His solution emphasizes sandboxing agents in secure environments rather than running them locally on machines with sensitive information.

### Shopify/liquid: Performance: 53% faster parse+render, 61% fewer allocations - (Simon Willison's Weblog)

https://simonwillison.net/2026/Mar/13/liquid/#atom-everything

# Summary

Shopify CEO Tobias Lütke used an AI-powered "autoresearch" system to achieve significant performance improvements to Liquid, Shopify's open-source Ruby template engine. The coding agent ran approximately 120 automated experiments across 93 commits, resulting in a 53% faster parse-and-render speed and 61% fewer memory allocations. Key optimizations included replacing regex-based tokenization with faster byte-scanning methods and pre-caching integer conversions.

The article highlights how this achievement demonstrates the effectiveness of AI-assisted programming: a robust test suite (974 unit tests) enabled the agent to safely experiment, the autoresearch pattern of iterative hypothesis testing proved highly productive, and clear benchmarking goals made optimization measurable. Notably, it shows how coding agents now allow busy executives to contribute meaningfully to development work—something Lütke's recent GitHub activity spike illustrates.

### AI should help us produce better code - (Simon Willison's Weblog)

https://simonwillison.net/guides/agentic-engineering-patterns/better-code/#atom-everything

# Summary

This article argues that AI coding agents should improve code quality rather than just accelerate development. The author contends that shipping worse code with agents is a deliberate choice—instead, teams should use agents to address technical debt by automating time-consuming but conceptually simple refactoring tasks (API redesigns, renaming, deduplication, file splitting). Since the cost of these improvements has dropped dramatically, maintaining code quality standards alongside feature development is now economically feasible.

Additionally, coding agents enable better decision-making by exploring multiple architectural approaches and running low-cost prototypes to validate technology choices. The author advocates for a "compound engineering loop" where teams document what works and continuously refine their agent instructions, allowing quality improvements to compound over time. This approach lets teams simultaneously ship features and maintain high code standards.

### Production query plans without production data - (Simon Willison's Weblog)

https://simonwillison.net/2026/Mar/9/production-query-plans-without-production-data/#atom-everything

# Summary

PostgreSQL 18 introduced new functions (`pg_restore_relation_stats()` and `pg_restore_attribute_stats()`) that allow developers to copy database statistics from production environments to development setups without needing to copy the actual data. This enables developers to replicate production query plans and test them locally, since the query planner relies on internal statistics that typically differ between environments. These statistics dumps are extremely compact—often under 1MB for databases with hundreds of tables—making this approach practical even for massive production databases.

Interestingly, SQLite has offered similar functionality for years through its writable `sqlite_stat1` and `sqlite_stat4` tables, allowing users to manually inject alternative statistics to control query planning. This feature is documented in SQLite's optimization guidelines and the `.fullschema` CLI command outputs both schema and statistics specifically to help reproduce query issues in testing without loading massive database files.

### Perhaps not Boring Technology after all - (Simon Willison's Weblog)

https://simonwillison.net/2026/Mar/9/not-so-boring/#atom-everything

# Summary

Simon Willison challenges the concern that LLMs will force developers toward "boring" mainstream technologies by limiting adoption of newer or less common tools. While this was a valid worry with earlier models, modern coding agents with improved context windows and agent frameworks appear to overcome this limitation. These agents can quickly learn unfamiliar tools by consulting documentation and examining existing code patterns, then successfully apply that knowledge to solve problems—even with private or cutting-edge libraries absent from training data.

In a follow-up note, Willison acknowledges that while LLMs may still exhibit biases in their technology recommendations (favoring popular tools like GitHub Actions and Stripe), the key finding is that human developers can choose whatever technology they prefer, and the agents will adapt and work effectively with it. The emerging "Skills" mechanism, where projects release official integrations for coding agents, further strengthens this capability.

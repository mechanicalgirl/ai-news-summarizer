### Watching for file changes on macOS - (Python Weekly)

https://alexwlchan.net/2026/watch-files-on-macos/

# Summary

The article explains how to detect file changes on macOS using the FSEvents API to build a live reload feature for a static site generator. The author walks through creating a Swift script that monitors specified directories for changes and prints notifications when files are modified, covering key concepts like setting up event streams, handling callbacks, and proper cleanup of system resources.

The implementation involves creating an FSEventStream with FSEventStreamCreate, scheduling it with a dispatch queue, and registering signal handlers to gracefully stop listening when the script exits. The author notes that FSEvents is ideal for this use case because it's designed to passively monitor large directory trees without constantly rescanning them, and discusses configuration options like latency settings for event coalescing and the importance of flushing stdout to ensure immediate output.

https://www.dash0.com/guides/opentelemetry-logging-python

# Summary

This guide explains how to integrate Python's standard logging module with OpenTelemetry to create a unified observability pipeline. Python's built-in logging produces structured output but remains disconnected from observability systems by default. The OpenTelemetry Python SDK includes a LoggingHandler that bridges this gap by converting standard Python LogRecords into OpenTelemetry-compliant log records, enabling correlation with traces, enrichment with resource attributes, and export via OTLP to observability backends without modifying existing logging calls.

The article provides a complete walkthrough for setting up this integration, starting with a practical example using a FastAPI application with JSON logging. It covers prerequisites (Python 3.14+, FastAPI, OpenTelemetry Collector), demonstrates how to configure the logging infrastructure using YAML, and shows how to wire the OpenTelemetry handler into Python's standard logging system for seamless integration with production observability backends.

### LLM Evaluation and AI Observability for Agent Monitoring - (Python Weekly)

https://blog.jetbrains.com/pycharm/2026/05/llm-evaluation-and-ai-observability-for-agent-monitoring/

# Summary

This article discusses the critical importance of LLM evaluation and AI observability for monitoring AI agents in production environments. As AI systems become increasingly complex—particularly multi-agent systems—organizations need rigorous monitoring and assessment. LLM evaluation determines whether an agent *can* work (testing basic capabilities), while agent observability ensures it *is* working (providing real-time visibility into internal processes and operational health). The article emphasizes that both are essential for success in real-world deployments.

The piece then details key evaluation metrics, focusing primarily on hallucination rate and toxicity scores. Hallucination rate measures factual accuracy, as hallucinations—outputs that seem convincing but lack factual support—significantly undermine user trust in AI systems. Toxicity scores identify harmful or offensive content, acting as a guardrail in public-facing applications. PyCharm's Hugging Face integration is highlighted as a practical tool that allows developers to discover and implement these evaluation models directly within their IDE, streamlining the process of building reliable AI agent systems.

### Django LiveView vs Phoenix LiveView: a real benchmark - (Python Weekly)

https://en.andros.dev/blog/80134668/django-liveview-vs-phoenix-liveview-a-real-benchmark/

# Summary

This article presents a real-world benchmark comparing Django LiveView and Phoenix LiveView by building identical alert dashboard applications in both frameworks. The results show that for typical operations (adding, deleting, and searching alerts), both frameworks perform nearly identically, with response times around 22ms for mutations and 7ms for searches. However, significant differences emerge in payload size and concurrency: Phoenix sends substantially smaller data packets (e.g., 67KB vs 327KB for large lists) because it automatically computes diffs, while Django LiveView requires developers to specify exactly which HTML selectors to update. At higher concurrency levels (25+ simultaneous clients), Phoenix scales notably better, with Django's latency roughly doubling Phoenix's at 50 concurrent users.

The author concludes that Django LiveView is a viable alternative to Phoenix LiveView for most use cases, with the main trade-offs being Django's simpler Python ecosystem versus Phoenix's superior scalability with large payloads and high concurrency. The choice depends on project needs: Django LiveView offers familiarity and explicitness for typical applications, while Phoenix excels when managing significant concurrent load or complex data updates.

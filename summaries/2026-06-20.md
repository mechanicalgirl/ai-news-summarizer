### The 2026 way of using importmaps in Django - (Python Weekly)

https://406.ch/writing/the-2026-way-of-using-importmaps-in-django/

# Summary

Matthias Kestenholz discusses the release of django-js-asset 4.0, which fundamentally changes how Django handles importmaps for JavaScript modules. The new approach abandons a single global importmap in favor of context-specific importmaps that are automatically assembled based on where they're needed—whether in Django admin, forms, or custom views. This addresses the problem of cluttered global importmaps when multiple packages use the feature, particularly with django-content-editor's new modular JavaScript implementation.

The article explains that importmaps solve Django's cache-busting challenge by mapping stable import names to hashed filenames. While the upgrade may require some configuration changes depending on usage, the new system integrates seamlessly with Django's existing media collection through the js_asset.Media class, which intelligently merges multiple importmaps and ensures they appear before other scripts. Kestenholz hopes to eventually standardize this approach in Django core, as browsers only support a single importmap per page.

### Plugins case study: Pluggy - (Python Weekly)

https://eli.thegreenplace.net/2026/plugins-case-study-pluggy/

# Summary

Pluggy is a Python library for building plugin systems, originally developed for pytest and later extracted as a standalone tool. It uses a decorator-based approach with "hooks"—functions that hosts expose and plugins implement. The article demonstrates Pluggy's usage through an htmlize tool example, showing how hosts define hooks using `HookspecMarker`, plugins implement them using `HookimplMarker`, and plugin discovery occurs via setuptools entry points or custom registration methods.

The author evaluates Pluggy against fundamental plugin concepts, noting that while plugin frameworks are easy to build, Pluggy offers valuable advanced features like automatic entry point registration, signature validation, consistent result collection, and plugin ordering. Whether adopting Pluggy is worthwhile depends on project needs and the tradeoff between adding a dependency versus implementing a custom solution.

### Profiling in PyTorch (Part 2): From nn.Linear to a Fused MLP - (Python Weekly)

https://huggingface.co/blog/torch-mlp-fusion

# Summary

This article explores PyTorch profiling at a deeper level by examining `nn.Linear` layers and Multi-Layer Perceptrons (MLPs). Building on Part 1's concepts, the authors demonstrate that `nn.Linear` is essentially a wrapper around matrix multiplication and bias addition, which are efficiently combined into a single GPU kernel using an "epilogue"—a technique that adds the bias during the matrix multiplication's final writeback to memory, avoiding redundant memory access. Notably, `torch.compile` provides minimal optimization for a single Linear layer since the operation is already fused, but becomes more valuable when multiple operations are stacked together.

The article further explains how PyTorch's transpose operation (`aten::t`) doesn't actually copy data but merely rewrites tensor metadata (shape and stride), allowing the kernel to read data in transposed order without overhead. The authors show that `torch.compile` can eliminate even this minimal CPU dispatch overhead by precomputing strides at compile time. They highlight how GPU kernels come in multiple precompiled variants (indicated by suffixes like "tn" in kernel names) to handle different input layouts, with the dispatcher selecting the appropriate kernel based on input strides—the GPU performs identical math regardless of whether a transpose appears in the CPU trace.

### Jupyter Enterprise Gateway: From Notebook to Kubernetes Cluster Admin - (Python Weekly)

https://www.elttam.com/blog/jupyter-enterprise-gateway

# Summary

This article details three critical security vulnerabilities discovered in Jupyter Enterprise Gateway (a distributed computing tool for Jupyter notebooks on Kubernetes) that allow unprivileged notebook users to escalate privileges and compromise entire Kubernetes clusters. The vulnerabilities were responsibly disclosed and patched in version 3.3.0, with each assigned a CVE. The core issue is that Enterprise Gateway holds elevated Kubernetes permissions and accepts user-controlled values that can be weaponized to bypass security restrictions.

The three vulnerabilities include: (1) a bypass of UID/GID restrictions through string padding that defeats validation checks, (2) a Jinja2 template injection enabling remote code execution, and (3) Kubernetes manifest injection through template rendering. The article demonstrates how an attacker can exploit these flaws to launch kernel pods running as root and execute arbitrary commands with Enterprise Gateway's elevated cluster privileges, potentially accessing secrets, mounting filesystems, and creating privileged pods across the entire cluster.

### How and why to run modified Python code using the ast module - (Python Weekly)

https://pydantic.dev/articles/eval-type-backport

# Article Summary

This article is a tutorial on using Python's `ast` module to parse, transform, and execute code at runtime. The author demonstrates how to parse Python code into an Abstract Syntax Tree (AST), manipulate it programmatically, and then compile and execute the modified code. The main example shows how the `eval-type-backport` library uses the `ast` module to convert modern Python syntax (like `int | str`) into older compatible syntax (like `Union[int, str]`) for use in earlier Python versions.

The tutorial walks through practical techniques including: subclassing `ast.NodeTransformer` to recursively modify AST nodes, using `ast.unparse()` to convert AST back to code strings, and compiling AST objects into executable code using the `compile()` function. The author emphasizes that this skill is increasingly valuable as AI becomes more prevalent, and demonstrates real-world applications in personal projects and the Pydantic Logfire SDK.

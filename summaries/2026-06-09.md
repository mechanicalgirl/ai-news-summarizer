### Pinterest Uses Content Fingerprints for URL Deduplication Across Millions of Domains - (InfoQ)

https://www.infoq.com/news/2026/06/pinterest-miqps-url-dedup/

# Summary

Pinterest engineers developed a data-driven URL normalization system called Minimal Important Query Param Set (MIQPS) to reduce duplicate content processing across its massive ingestion pipeline. Rather than relying on manual allowlists or domain-specific rules, MIQPS evaluates whether removing query parameters changes rendered page content—if changes exceed a threshold, the parameter is retained; otherwise it's classified as noise. This approach solves the scalability problem of handling millions of heterogeneous domains with inconsistent URL structures.

The system operates by collecting URLs, rendering pages, generating content fingerprints, and analyzing parameter importance offline before applying precomputed rules at runtime. By separating expensive analysis from runtime processing and using early-exit logic to avoid unnecessary renders, MIQPS significantly reduces infrastructure overhead from duplicate URL variants while maintaining data accuracy. Pinterest notes that this data-driven approach is more reliable than traditional methods like canonical tags, which are often missing or polluted with tracking parameters.

The discussion emphasizes that proper supervision should be governed by risk assessment—balancing the probability of AI success, the impact of failures, and error detectability—while acknowledging that the AI-native development space is still in its early "forming" stage as developers find the right equilibrium between autonomy, risk, and code quality.

### Terraform 1.15 Closes Gap to OpenTofu on Dynamic Sources and Deprecation - (InfoQ)

https://www.infoq.com/news/2026/06/terraform-1-15/

# Summary

Terraform 1.15, released by HashiCorp on April 29, 2026, introduces several significant features that close gaps with its open-source competitor OpenTofu. The most notable addition is dynamic module sources, allowing variables to be used in module source and version attributes through a new `const` attribute on variables—a capability OpenTofu introduced nearly two years earlier in version 1.8.0. The release also adds formal deprecation mechanisms for variables and outputs with warning messages, a new `convert()` function for inline type conversion, type constraints for output blocks, and native Windows ARM64 support.

While Terraform 1.15 addresses long-standing community requests, the two projects have diverged significantly since OpenTofu forked from Terraform in August 2023. OpenTofu has focused on features like native end-to-end state encryption, for_each on provider blocks, OCI registry support, and OpenTelemetry tracing, while Terraform has prioritized ephemeral values, write-only attributes, and HashiCorp Cloud Platform integration. This release demonstrates HashiCorp's effort to reclaim feature parity on key functionality that users have requested for years.

### Zero Reaches 1.0, Marking the First Stable Release of Rocicorp's Web Sync Engine - (InfoQ)

https://www.infoq.com/news/2026/06/zero-version-1/

# Summary

Rocicorp has released Zero 1.0, the first stable version of its web synchronization engine after nearly two years of development. The release is largely symbolic—functionally similar to the previous 0.26.2 version with only minor updates like a schema change hook for Supabase and various bug fixes. The milestone signals API stability and Rocicorp's commitment to maintaining Zero with minimal future breaking changes.

Zero works by syncing a local cache replica of a Postgres database, allowing queries to return results instantly from the client-side cache while updates sync from the server in the background. Community reception has been positive regarding developer experience but cautious about production readiness, with noted limitations including Postgres-only support, lack of view syncing, unsupported column types, and a 718 KB uncompressed client bundle. The platform competes in a crowded field alongside ElectricSQL, InstantDB, PowerSync, and Convex, with server-side rendering still on the roadmap.

### Long-Running Agents - (Radar)

https://www.oreilly.com/radar/long-running-agents/

# Summary

This article from O'Reilly, written by Addy Osmani, introduces "long-running agents"—AI systems that can maintain progress on tasks over extended periods (hours, days, or weeks) across multiple sessions and sandboxes, unlike traditional chat-based AI agents that operate within a single context window. Long-running agents can persist state outside the model's context, recover from failures, and resume work by leaving behind structured artifacts, solving key limitations of current agents that forget context, prematurely declare tasks complete, or reintroduce previously-fixed bugs.

The article clarifies that "long-running" encompasses three related but distinct concepts: long-horizon reasoning (multi-step planning and coherence), long-running execution (processes that invoke models thousands of times over extended periods), and persistent agency (agents with identity and accumulated memory across tasks). Osmani argues this shift matters economically because it makes previously infeasible delegations possible and outlines how engineers can implement long-running agents today without building everything from scratch.

### I Let an AI Agent Run 40 Experiments While I Slept - (Radar)

https://www.oreilly.com/radar/i-let-an-ai-agent-run-40-experiments-while-i-slept/

# Summary

The author set up an AI agent to autonomously run machine learning experiments overnight, inspired by Andrej Karpathy's autoresearch project. The agent successfully completed 40 experiments, improving validation loss by 5.9% and reducing memory usage from 44GB to 17GB. However, the experiment revealed critical failures: a linter bug caused the agent to waste four hours chasing a false lead without flagging the issue, and the agent lacked the ability to distinguish between legitimate improvements and subtle regressions.

The author also tested a similar autonomous agent pattern on fixing broken Claude Code skills in parallel, which yielded mixed results—while 13 of 15 skills improved, three experienced subtle regressions that went undetected, including an agent removing a necessary safeguard and another making descriptions too precise for real-world fuzzy user queries. The key takeaway is that while AI agents excel at systematically exploring possibilities, they struggle with context-dependent decision-making and require human oversight to catch non-obvious failures and unintended consequences.

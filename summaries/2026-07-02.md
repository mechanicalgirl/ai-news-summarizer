### Elastic Open-Sources Atlas Agent Memory Based on Cognitive Science - (InfoQ)

https://www.infoq.com/news/2026/06/elastic-atlas-agent-memory/

# Summary

Elastic has open-sourced Atlas, a cognitive science-based memory system for AI agents built on Elasticsearch that addresses the challenge of managing long interaction histories. Rather than overloading an LLM's context window with entire conversation histories—which is costly, slow, and prone to the "lost in the middle" effect—Atlas maintains three separate memory types inspired by cognitive science: episodic (what happened), semantic (facts), and procedural (what works). The system automatically consolidates memories through LLM-driven analysis, converting episodic events into durable semantic facts with supporting evidence and updating procedural playbooks based on success rates.

Atlas retrieves relevant memories using a hybrid approach combining lexical search (BM25) and semantic search (Jina v5), merged with Reciprocal Rank Fusion and reranked using a cross-encoder. The system integrates with agents via the Model Context Protocol (MCP) and ensures user privacy through document-level security. In testing, Atlas achieved a 0.89 Recall@10 score on question-answering tasks, demonstrating its effectiveness as a scalable long-term memory solution for AI agents that can persist across years of interactions.

### AWS Launches Lambda MicroVMs for Isolated Agent and User Code Execution - (InfoQ)

https://www.infoq.com/news/2026/06/aws-lambda-microvms/

# Summary

AWS has launched Lambda MicroVMs, a new serverless compute service that runs each user session or AI agent in its own isolated Firecracker virtual machine with hardware-level isolation and snapshot-based rapid launch. Available in five regions with up to 16 vCPUs and 32GB memory, MicroVMs address a gap in cloud computing by combining the strong isolation of VMs, near-instant launch times of containers, and stateful execution capabilities—eliminating previous tradeoffs that forced developers to choose between security, speed, or state preservation.

The service targets multi-tenant applications running untrusted code, particularly AI-generated code that cannot be audited. Unlike Lambda Functions designed for event-driven workloads, MicroVMs use a suspend/resume lifecycle that preserves application state across idle periods, making them practical for interactive sessions. While Lambda MicroVMs doesn't enable entirely new use cases, it improves the cost/performance/security tradeoff compared to existing options like Cloudflare Sandboxes, Google's GKE Agent Sandbox, and Azure Container Apps, each of which makes different architectural tradeoffs.

### Article: Scaling Java-Based Real-Time Systems: The Hidden Tradeoffs of Event-Driven Design - (InfoQ)

https://www.infoq.com/articles/tradeoffs-event-driven-design/

# Summary

This article examines the hidden challenges of scaling Java-based real-time systems using event-driven architecture with Apache Kafka. Based on the author's experience building a cloud contact center platform handling 80,000+ concurrent calls, the piece argues that while event-driven design offers benefits like loose coupling and scalability, it creates critical tradeoffs for systems where real-time responsiveness is essential. The fundamental tension is that asynchronous event processing introduces unacceptable latency for critical paths like call signaling and agent state transitions, where eventual consistency is functionally equivalent to system failure.

The article identifies several production issues including Kafka event replay causing startup delays that disable Kubernetes autoscaling, RocksDB compaction creating unpredictable latency spikes, and synchronous REST calls within Kafka consumers cascading into hours of consumer lag. The author concludes that event-driven architecture works well when eventual consistency is acceptable, but real-time communication systems require synchronous or near-synchronous communication for critical paths rather than purely asynchronous pipelines, highlighting architectural tradeoffs that diagrams rarely reveal.

### Eliya 25 Brings a JVM-Level Diagnostic Profile to OpenJDK 25 LTS - (InfoQ)

https://www.infoq.com/news/2026/06/eliya-jvm-diagnostic-profile/

# Summary

Asymm Systems has released Eliya 25.0.3, an OpenJDK 25 LTS distribution that introduces a JVM-level diagnostic profile for production environments through the flag `-XX:EliyaProfile=Production`. Designed for Java teams requiring reliable crash, memory, and runtime diagnostics—particularly in regulated industries—Eliya consolidates existing HotSpot features to configure heap dumps on out-of-memory errors, enable exit-on-OOM, enable Native Memory Tracking, ensure predictable crash log locations, and unlock diagnostic options for JFR sampling and profiler attachment. The distribution maintains full compatibility with upstream OpenJDK 25 while adhering to OpenJDK's Java APIs and behavior standards.

Eliya follows a philosophy that most production policies should be managed externally through Kubernetes, service meshes, or wrapper scripts, with JVM-level policy reserved only when necessary for inaccessible internal behavior or when external systems cannot sanitize sensitive data before it's written to disk. Currently in Phase 1, the Production profile mainly establishes a policy control point for future enhancements, with Phase 2 planned for late 2026 to include bundled diagnostic tools, FIPS variants, continuous JFR recording, and unified GC logging. Support extends through September 2029, with critical patches released within 2 weeks of upstream OpenJDK updates.

# Summary

This article discusses the decline of "tokenmaxxing"—the practice of wastefully consuming AI tokens to appear productive—as AI providers shift toward more sustainable pricing models. This led to inevitable price increases across the industry, finally making the true cost of AI visible to users and bringing an end to the era of wasteful token consumption.

### Beyond Prompt Injection - (Radar)

https://www.oreilly.com/radar/beyond-prompt-injection/

# Summary

The article argues that current AI security approaches are fundamentally flawed because they focus on preventing prompt injection attacks at the input layer, when the real threat lies in what AI agents do with compromised instructions. Using examples like the ForcedLeak vulnerability in Salesforce's Agentforce—where malicious instructions hidden in a form field caused an AI agent to exfiltrate data to a trusted domain—the author demonstrates that traditional input filtering and prompt hardening are insufficient defenses.

The core issue is that language models process all information as an undifferentiated token stream, making it structurally impossible to completely prevent injection attacks. Rather than continuing to focus on making models "unfoolable," organizations should shift their security model to assume compromise will occur and instead implement controls around what actions agents can take. This requires a fundamental rethinking of AI security that prioritizes limiting agent capabilities and enforcing strict permission boundaries, rather than trying to prevent attackers from manipulating model inputs.

### Agent Memory - (Radar)

https://www.oreilly.com/radar/agent-memory/

# Agent Memory Summary

Agent memory represents a critical challenge in AI development because large language models (LLMs) are stateless by design—they have no inherent memory between interactions. While chat interfaces appear to remember previous messages, they actually resend the entire conversation history with each prompt. The article explores how companies and developers are working to solve this problem, as true agent memory would enable AI agents to build on past learnings, adapt to users, resume work after restarts, and maintain continuity.

The article distinguishes between different types of agent memory beyond simple conversation history. Conversational memory stores message exchanges but becomes unwieldy as conversations grow long, filling prompts with irrelevant context. Semantic memory, conversely, stores durable facts and meanings that should persist across conversations—such as user preferences or system capabilities—rather than exact wording. Effective agent memory systems require decisions about what to store, where to store it, how to retrieve it, and when to summarize or compress it, making memory engineering far more complex than simply appending prior messages to prompts.

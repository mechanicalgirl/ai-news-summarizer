### An announcement from the Steering Council regarding the JIT project - (Python Weekly)

https://discuss.python.org/t/an-announcement-from-the-steering-council-regarding-the-jit-project/107638

# Summary

The Python Steering Council has announced that the experimental JIT (just-in-time) compiler in CPython must now undergo formal review through the PEP (Python Enhancement Proposal) process before it can be considered a permanent, supported feature. While praising the technical work done by developers, the Council notes that the JIT was merged into the main branch as an experiment without adequately addressing key questions about long-term maintenance, compatibility with existing CPython features like debuggers and profilers, success metrics, and relationships to other JIT implementations. The Council is requesting a comprehensive Standards Track PEP that addresses these concerns and sets a six-month deadline for submission and resolution.

Until such a PEP is accepted, no new JIT features, optimizations, or performance work can be added to the main branch, though bugfixes and security fixes may continue. The Council emphasizes this is not meant to shut down the project but to give it the clarity and explicit community commitment that a change of this magnitude deserves. If no acceptable PEP is approved within six months, the JIT code would need to be removed from the main branch and developed separately.

### The Smallest Brain You Can Build - (Python Weekly)

https://ranpara.net/posts/perceptron-explained-from-scratch/

# Summary

This article introduces the perceptron, the foundational building block of modern neural networks, by explaining how it works through simple, intuitive examples rather than complex mathematics. The perceptron mimics how a neuron works by taking inputs, multiplying them by weights, adding a bias, and producing a yes-or-no output based on whether the result exceeds zero. The author walks through practical examples—from determining if a number is positive to predicting student pass/fail based on exam scores—demonstrating how the perceptron learns by adjusting its weight and bias when it makes mistakes.

The article covers essential concepts including the decision boundary (where the perceptron flips from false to true), the importance of bias in moving that boundary away from zero, and practical training details like epochs and learning rate. It also explains why normalizing data (scaling inputs to a consistent range) improves training efficiency by preventing large input values from causing erratic weight updates. The author provides complete, readable Python code and interactive demos, emphasizing that while a single perceptron can only draw one straight line to separate data, stacking multiple neurons together creates the neural networks that power modern AI.

### Are you really expected to run five type-checkers now? - (Python Weekly)

https://pyrefly.org/blog/too-many-type-checkers/

# Summary

The article argues that Python library maintainers should prioritize running multiple type-checkers on their **test suites** rather than their internal source code. This approach is more valuable because tests validate how the public API behaves when users interact with it—what users actually care about—whereas checking internal code mainly validates implementation details. The author illustrates this with Polars' experience, showing how satisfying multiple type-checkers in source code creates messy code polluted with conflicting type-ignore comments, while the same function's test passes cleanly across all type-checkers.

The proliferation of type-checkers (mypy, Pyrefly, Pyright, ty, Zuban) exists because the typing specification has ambiguous areas where different checkers make different trade-offs between strictness and leniency. Rather than overwhelming maintainers by requiring all checkers on source code, the recommendation is to run as many as feasible on tests to ensure compatibility across users' preferred type-checking tools.

### Ordered key sharding in DynamoDB - (Python Weekly)

https://death.andgravity.com/albumtitle

# Summary

This article explores solutions for maintaining a sorted index in DynamoDB when a single partition can't handle the required throughput. The author starts with a music database example requiring 10,000 queries/second with sorted results, which exceeds DynamoDB's per-partition limits by 21x. While a sparse index or single partition with a sort key would be ideal, the throughput requirements force the use of sharding.

The article evaluates several sharding strategies—random suffixes, hash-based sharding, and first-character sharding—each with tradeoffs between throughput and ordering. The key insight is that using prefix-based sharding with evenly-distributed boundaries works best: by splitting sorted album titles into N equal ranges and using a long enough prefix (around 20 characters) as the shard ID, you can maintain alphabetical ordering across shards while distributing load evenly. This approach elegantly avoids the complexity of tries or prefix trees by simply dividing the dataset into equal-sized contiguous ranges.

### Running Python code in a sandbox with MicroPython and WASM - (Python Weekly)

https://simonwillison.net/2026/Jun/6/micropython-in-a-sandbox/

# Summary

Simon Willison describes his new **micropython-wasm** package, an alpha-stage Python sandbox solution that executes untrusted code safely within Python applications. He developed this to address security concerns with his plugin systems (Datasette, LLM, sqlite-utils), which currently run plugins with full privileges. The solution combines MicroPython—a lightweight Python implementation designed for constrained environments—compiled to WebAssembly and executed via the actively-maintained wasmtime library, which provides memory and CPU limits, controlled file/network access, and host function integration.

The implementation involved using AI coding assistants to compile MicroPython to WASM and solve technical challenges like maintaining persistent interpreter state across multiple code executions and supporting host functions. The final 362KB WebAssembly blob enables safe code execution with features like fuel-based CPU limits and configurable resource restrictions. Willison acknowledges the irony of creating a new sandbox library after criticizing unmaintained ones, but emphasizes that WebAssembly's worst-case failure is an exception rather than system compromise, making it an acceptable risk for his use case.

### Cangjie, a New Open-Source Compiled Language with Native Effect Handlers and Algebraic Data Types - (InfoQ)

https://www.infoq.com/news/2026/05/cangjie-effect-handlers-adt/

# Summary

Cangjie (CJ) is a new open-source, compiled programming language developed by Huawei's Programming Languages Lab and positioned as an alternative to Java, Kotlin, and Swift. It features static typing, pattern matching, algebraic data types, and compiles to machine code across multiple platforms including Linux, macOS, Windows, Android, iOS, and HarmonyOS.

The language's most significant innovation is its native support for effect handlers—a feature that generalizes exceptions and simplifies dynamic binding. Effect handlers allow code to interact with its calling context through `perform` and `resume` keywords, enabling elegant solutions for tasks like logging, caching, memoization, and dependency injection. This approach is particularly powerful for cross-platform scenarios where the same library needs to adapt its behavior (such as logging) based on the device's capabilities, without requiring complex workarounds found in traditional exception-handling languages.

### Netflix Serves 84% of Query Results from Cache with Interval-Aware Caching in Apache Druid - (InfoQ)

https://www.infoq.com/news/2026/05/netflix-druid-interval-cache/

# Summary

Netflix has developed an interval-aware caching strategy for Apache Druid that achieves an 84% cache hit rate for analytics queries while reducing load on the database by 33%. The solution addresses a key inefficiency in rolling window dashboards, where near-identical queries with slightly shifted time ranges are treated as separate requests by traditional caching systems, leading to redundant computation and repeated data scans across Druid's massive datasets.

The system works by decomposing query results into time-aligned segments rather than caching full outputs. It reuses cached segments for stable historical portions of time windows while only recomputing the most recent interval from Druid, then merging results. Operating as an external proxy layer, this approach delivers significant performance improvements including 66% faster P90 query times and up to 14x reduction in result bytes for some workloads. Netflix plans future enhancements including direct integration into Apache Druid itself and support for templated SQL queries.

The shift to serverless allowed the team to focus on application code rather than infrastructure management, fundamentally addressing the mismatch between business growth and the original architecture's limitations.

### Article: Local-First AI Inference: A Cloud Architecture Pattern for Cost-Effective Document Processing - (InfoQ)

https://www.infoq.com/articles/local-first-ai-inference-cloud/

# Summary

This article presents the "Local-First AI Inference" pattern, a three-tier cloud architecture designed to reduce costs and improve efficiency in AI-powered document processing. Rather than sending all documents to cloud AI endpoints, the pattern routes 60-80% of structurally predictable documents (like engineering drawings or invoices) through fast, cost-free local deterministic extraction first. Only documents that fail local processing criteria are sent to cloud AI services like Azure OpenAI, reducing API costs by 75% and processing time by 55%.

The key architectural insight is that the most important decision isn't which AI model to use, but whether to call an AI model at all. The pattern implements three tiers: local deterministic processing for routine cases, cloud AI for complex edge cases, and human review to bound error rates and prevent silent hallucinations. Testing on 4,700 engineering PDFs demonstrated this approach reduced costs from $47 to $10-15 while maintaining reliability—far superior to both cloud-only approaches (which risk hallucinations) and local-only methods (which miss scanned documents entirely).

https://www.infoq.com/news/2026/05/cloudflare-dynamic-workflows/

# Summary

Cloudflare has released Dynamic Workflows, an MIT-licensed library that enables workflow code to differ for each tenant, agent, or request at runtime, removing the previous constraint of requiring fixed workflow code at deployment. The ~300 line TypeScript library uses a Worker Loader to route execution to the correct tenant's code, allowing platforms to support AI-generated workflows, CI/CD pipelines, and agent-based systems where each has its own dynamic code while maintaining full durable execution semantics including retries, hibernation, and event waiting.

The release represents part of Cloudflare's broader platform strategy to make all worker bindings dynamically dispatchable per tenant with zero idle cost. For agents specifically, this allows them to write their own executable plans with independently retryable steps.

### From Capabilities to Responsibilities - (Radar)

https://www.oreilly.com/radar/from-capabilities-to-responsibilities/

# Summary

This article argues that high-stakes AI systems (those capable of moving money, changing infrastructure, or modifying critical records) must shift from a capabilities-based design model to a responsibility-based one. The author criticizes the current industry reliance on Human-in-the-Loop (HITL) approval systems, which become operational bottlenecks at scale—leading to alert fatigue where humans approve decisions without proper review rather than providing meaningful governance.

The proposed solution is "Governance by Exception," where humans establish policy upfront and the runtime automatically enforces it, with only truly exceptional cases escalated for human review. Rather than asking "What can this agent do?" (capabilities frame), the focus should shift to "What is this agent authorized to do?" (responsibility frame), mirroring role-based access control in traditional software. This represents a fundamental change in how AI systems are governed for production environments.

### Fighting Tool Sprawl: The Case for AI Tool Registries - (Radar)

https://www.oreilly.com/radar/fighting-tool-sprawl-the-case-for-ai-tool-registries/

# Summary

As enterprise AI agent adoption accelerates, organizations face a critical infrastructure problem: the absence of centralized tool registries is creating duplicated engineering efforts, security vulnerabilities, and operational blindness. Individual teams are building AI tools in ad hoc, undocumented ways across the organization, leading to fragmentation, missed security reviews, and unclear system dependencies—a coordination failure that treats an infrastructure problem as an application-layer issue.

The solution is enterprise-level shared tool registries that are internal, scoped to each organization's specific regulatory, security, and operational needs. Similar to how package managers (npm, PyPI) solved code duplication decades ago, AI tool registries would enable teams to discover, depend on, and govern shared tools while providing visibility and risk management. Evidence from security surveys shows that without this infrastructure, most organizations lack full security approval for their agents and experience high rates of security incidents, making centralized registries foundational rather than optional as AI agent deployments scale beyond early experiments.

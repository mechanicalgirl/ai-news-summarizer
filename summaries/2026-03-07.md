### Deprecate confusing APIs like “os.path.commonprefix()” - (Python Weekly)

# Summary

Seth Larson argues that Python's `os.path.commonprefix()` function should be deprecated due to its confusing behavior and history of causing security vulnerabilities. The function, present in Python for 35 years, compares strings character-by-character rather than by path segments, misleading users into thinking it safely identifies path prefixes. This confusion led to multiple security issues, including vulnerabilities in pip, SecureDrop, and the tarfile module, despite being documented since 2002.

Larson proposes that APIs causing frequent accidental misuse—especially security-relevant ones—should be prioritized for deprecation over strict backwards compatibility. He emphasizes that documentation alone is insufficient; instead, deprecation, renaming, and static analysis tools are needed to eliminate such "footguns." The author has already submitted pull requests to add explicit security warnings and deprecate the function in Python 3.15, hoping this case will accelerate future deprecations of confusing or insecure APIs.

### Invent your own comprehensions in Python - (Python Weekly)

https://www.pythonmorsels.com/custom-comprehensions/

# Summary

While Python has built-in list, set, and dictionary comprehensions, it lacks comprehensions for tuples, frozensets, and other collection types. However, you can effectively "invent" your own comprehensions by pairing generator expressions with iterable-accepting functions and classes. For example, `tuple(n**2 for n in numbers)` creates a tuple comprehension-like syntax, and `frozenset(w.strip("-.,!?\"'") for w in verse.lower().split())` creates a frozenset comprehension-like structure.

Generator expressions work particularly well with functions that accept iterables, such as collection constructors (tuple, frozenset, Counter) and reducer functions (sum, any, all). This technique allows developers to create custom comprehension-like syntax for any iterable-accepting callable, providing a flexible workaround for Python's limited built-in comprehension syntax.

### Python Type Checker Comparison: Empty Container Inference - (Python Weekly)

https://pyrefly.org/blog/container-inference-comparison/

# Summary

Python type checkers face a fundamental challenge when inferring types for empty containers like `[]` and `{}`, since they contain no information about what types should be stored inside. The article examines three distinct strategies different type checkers use to solve this problem: (1) **Inferring `Any` type** (used by Pyre, Pyright, and Typy), which is simple and efficient but sacrifices type safety and allows bugs to slip through; (2) **Inferring from all usages** (Pytype's approach), which creates a union of all types used across the container and catches errors at read-time, but error messages may be far removed from where the actual bug occurs; and (3) **Inferring from only the first usage** (Mypy and Pyrefly's approach), which uses the first operation on the container to guess its type, providing error messages closer to the bug location while still allowing developers to override incorrect guesses with explicit annotations.

Each strategy involves trade-offs between type safety, performance, and the usefulness of error reporting. Strategy 3 offers a practical middle ground by catching bugs at their source without burdening developers with unnecessary type annotations everywhere, making it the most pragmatic approach for real-world code development.

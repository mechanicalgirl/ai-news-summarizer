### VoidZero’s Experimental Oxc Angular Compiler with Up to 20x Faster Build Performance - (InfoQ)

https://www.infoq.com/news/2026/04/angular-compiler-rust/

# Summary

VoidZero has released an experimental Rust-based Angular compiler built on its Oxc toolchain that achieves dramatically faster build times—up to 20.7x faster than the current Angular CLI compiler on some codebases. The compiler is distributed as a Vite plugin with full Hot Module Replacement support and works by implementing template compilation natively in Rust rather than converting templates to TypeScript for compilation, significantly reducing overhead.

While the project was developed with AI coding agents (Claude Code and Codex) alongside experienced engineers over two months, it is released for research purposes only with no plans for ongoing maintenance. The Angular team has acknowledged interest in next-generation compilers and indicated plans to explore similar performance improvements, though template type checking support is unlikely in this experimental version.

### AWS Interconnect Reaches General Availability with Managed Multicloud and Last-Mile Connectivity - (InfoQ)

https://www.infoq.com/news/2026/04/aws-interconnect-multicloud-ga/

# Summary

AWS has announced the general availability of AWS Interconnect, a managed private connectivity service with two main capabilities: Interconnect multicloud for Layer 3 private connections between AWS VPCs and other cloud providers (starting with Google Cloud, with Azure and Oracle Cloud coming later in 2026), and Interconnect last mile for simplified high-speed connections from branch offices and data centers to AWS. The service dramatically reduces provisioning time from weeks or months to minutes by abstracting away physical infrastructure complexity and allowing on-demand provisioning through the AWS console.

This collaboration with Google Cloud is architecturally significant as the first fully managed multicloud interconnect between two major hyperscalers. AWS has published the underlying specification on GitHub under an open license, positioning it as an industry standard and pressuring other cloud providers to adopt the approach. Connections are secured with MACsec encryption and routed entirely over private networks, never traversing the public internet. However, practitioners should be aware of technical considerations including non-overlapping IP address ranges, MTU matching requirements between clouds, and current limitations like the lack of Google Cloud web console provisioning at GA.

### GitHub Targets Large Merge Problem with Stacked PRs - (InfoQ)

https://www.infoq.com/news/2026/04/github-stacked-prs/

# Summary

GitHub has launched a native stacked pull request (PR) workflow through a new CLI extension called gh-stack to address the problem of large, difficult-to-review pull requests. Stacked PRs are a code-review pattern where each branch targets the previous branch in a series rather than the main branch, allowing developers to continue working on later layers while earlier ones are under review. Research shows PRs between 200-400 lines have 40% fewer defects and are approved three times faster than larger ones, making this approach particularly effective for breaking down large features into manageable chunks.

The gh-stack extension automates the traditionally manual mechanics of maintaining stacked workflows, including cascading rebases and atomic force-pushes across the stack. Key features include a visual stack map within GitHub's PR interface for navigation, AI agent integration to automatically break diffs into layers, and enforcement of CI and branch protection rules as if each PR targeted main directly. While the stacked diff pattern is not new—having been used by Meta and Google for nearly a decade—GitHub's integration directly into its platform is notable. However, community reaction has been mixed, with concerns raised about merge strategy limitations, cognitive overhead, and whether the workflow truly makes sense on GitHub's infrastructure, particularly compared to existing competitors like Graphite.

### How Slack Manages Context in Long-Running Multi-agent Systems - (InfoQ)

https://www.infoq.com/news/2026/04/slack-agent-context-management/

# Summary

Slack engineers developed a structured approach to managing context in long-running multi-agent systems, moving away from simply accumulating chat logs. They implemented three complementary "context channels": a director's journal (storing structured working memory), a critic's review (storing validated findings with credibility scores), and a critic's timeline (maintaining chronological, corroborated findings). This coordinator/dispatcher design assigns specialized roles—experts generate findings, critics evaluate them for accuracy and assign credibility scores, and the director makes strategic decisions based on validated information.

Rather than passing all information at every step, Slack's approach builds structured summaries that agents can reliably build upon while preventing hallucinations and maintaining coherence across hundreds of requests. This strategy addresses the fundamental challenge that long-running agent sessions quickly exceed LLM context windows, degrading response quality. The system's success demonstrates a broader principle: filtering information through validation layers and preserving only credible, non-duplicate evidence creates a foundation that specialized agents can effectively work with while maintaining system coherence.

### GitHub Uses eBPF to Eliminate Deployment Risks and Prevent Circular Failures - (InfoQ)

https://www.infoq.com/news/2026/04/github-ebpf-deployment/

# Summary

GitHub has implemented an innovative solution using eBPF (extended Berkeley Packet Filter) to detect and prevent circular dependencies during deployments, which can cascade and prolong outages when deployment tools rely on the services they're meant to fix. By running custom programs at the Linux kernel level, GitHub monitors deployment processes within controlled environments (cGroups) and can selectively block risky network requests based on domain names through DNS-aware filtering. This shifts circular dependency detection from a reactive, incident-driven process to proactive identification, improving deployment safety and mean time to recovery.

The solution represents a broader industry trend toward kernel-level observability and control for managing increasingly complex systems. Beyond preventing circular dependencies, the approach provides deployment auditing and resource limit enforcement. Similar strategies are being adopted by other large-scale platforms like Google (through hermetic builds), AWS (via cell-based architecture), and cloud-native projects like Kubernetes and Cilium, reflecting a growing recognition that systems must be designed to ensure recovery paths remain independent of the systems they're meant to fix.

### Presentation: Week-Long Outage: Lifelong Lessons - (InfoQ)

https://www.infoq.com/presentations/outage-lessons/

# Summary

Molly Struve, a Staff Site Reliability Engineer at Netflix, recounts a catastrophic six-day outage at her previous company, Kenna Security, that occurred during a March 2017 Elasticsearch upgrade. The outage began when upgrading a critical 21-node Elasticsearch cluster (from version 2 to 5)—which was the cornerstone of their cybersecurity platform—caused unexpected CPU and load spikes. Despite initial assumptions that the cluster would self-balance overnight, nodes began crashing the following morning, escalating into a week-long crisis.

Beyond the technical details of the outage, Struve emphasizes that the most valuable lessons learned were not purely technical but human-centered. She highlights the importance of practices like Failure Mode and Effects Analysis (FMEAs), shadow traffic testing, and regularly exercising rollback mechanisms. However, she stresses that psychological safety—built through widening communication circles early in incidents and having leadership that actively defends the team—proved equally critical to organizational resilience and preventing similar outages in the future.

### Legare Kerrison and Cedric Clyburn on LLM Performance and Evaluations - (InfoQ)

https://www.infoq.com/news/2026/04/kerrison-clyburn-llm-performance/

# Summary

Legare Kerrison and Cedric Clyburn from Red Hat discussed practical methods for evaluating and optimizing Large Language Model (LLM) performance at the Arc of AI 2026 Conference. They emphasized that while generic leaderboards exist, teams must evaluate LLMs based on their specific business requirements rather than generic benchmarks. The speakers highlighted a key "tradeoff triangle" in LLM deployment: optimizing for any two of the three factors—model quality (accuracy), responsiveness (latency), and cost—negatively impacts the third.

To address this challenge, the speakers recommend defining clear Service Level Objectives (SLOs) with specific performance metrics tailored to application needs. Critical metrics include Requests Per Second (RPS) for throughput, Time to First Token (TTFT) for perceived latency, and Inter-Token Latency (ITL) for streaming responsiveness. Different use cases require different targets—for example, an e-commerce chatbot needs TTFT ≤200ms and ITL ≤50ms, while RAG applications prioritize accuracy over speed with more relaxed latency requirements. Hardware optimization techniques like structured generation, speculative decoding, and caching can further improve efficiency, with teams potentially running LLMs locally when practical.

### When Correct Systems Produce the Wrong Outcomes - (Radar)

https://www.oreilly.com/radar/when-correct-systems-produce-the-wrong-outcomes/

# Summary

This article examines how AI systems can produce incorrect outcomes even when all individual components function correctly. The core problem is "behavioral drift"—where autonomous systems remain operational but gradually diverge from their intended trajectory without triggering alerts. In continuously operating AI systems that combine retrieval, reasoning, and tool invocation, the traditional assumption that "correct components equal a correct system" breaks down because each step (retrieving data, reasoning about it, taking action) may be locally valid while the overall sequence produces globally misaligned results.

The fundamental issue is a "composability problem": correctness no longer composes cleanly in autonomous systems. Unlike discrete, bounded operations in traditional software, continuously operating AI systems generate emergent behavior over time where each reasonable individual decision can reinforce problematic interpretations. This means exhaustive testing of individual components cannot guarantee correct system behavior, revealing a critical limit in current observability and testing approaches for modern AI systems.

### Show Your Work: The Case for Radical AI Transparency - (Radar)

https://www.oreilly.com/radar/show-your-work-the-case-for-radical-ai-transparency/

# Summary

The article argues for radical transparency in how we use AI, advocating that professionals should share their complete AI interactions—prompts, responses, iterations, and all—rather than just polished final outputs. Drawing on Dorothy Leonard's concept of "deep smarts," the author contends that showing the full conversational process reveals the human judgment and expertise involved, making collaborators better able to evaluate the work and trust the results. Hiding the AI process makes it invisible and ultimately erodes trust.

The author challenges the common instinct to clean up AI interactions before sharing work, arguing this approach is actually counterproductive and defensive. By showing the messy back-and-forth of AI collaboration, including dead ends and refinements, professionals demonstrate their judgment and expertise rather than appearing to have outsourced their thinking to a machine. This transparency allows teams and stakeholders to see where human expertise ends and AI pattern-matching begins, ultimately building stronger trust and preserving institutional knowledge.

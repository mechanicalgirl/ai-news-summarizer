### QCon London 2026: Rewriting All of Spotify's Code Base, All the Time - (InfoQ)

https://www.infoq.com/news/2026/03/spotify-honk-rewrite/

# Summary

At QCon London 2026, Spotify engineers presented "Honk," an AI-powered coding agent that performs large-scale code migrations across the company's entire codebase. Moving beyond deterministic scripts that could only handle 70% of migrations, Honk uses LLMs to tackle complex edge cases, achieving 1,000 merged pull requests every 10 days. The system packages the entire software development process—requirements, code generation, building, testing, and iteration—with a critical architectural decision to separate agent runtime from verification runtime for scalability across hundreds of repositories.

As Honk's code generation capacity has scaled dramatically, the bottleneck has shifted from writing code to reviewing pull requests. Spotify is addressing this through three strategies: cultural changes around review expectations, tooling improvements like PR prioritization, and most importantly, codebase standardization. The team argues that standardized codebases produce simpler, more correct agent code that's easier to review, creating a virtuous cycle that increases capacity and drives further standardization.

### HubSpot’s Sidekick: Multi-Model AI Code Review with 90% Faster Feedback and 80% Engineer Approval - (InfoQ)

https://www.infoq.com/news/2026/03/hubspot-ai-code-review-agent/

# Summary

HubSpot introduced Sidekick, an AI-powered code review agent that analyzes pull requests and provides automated feedback to developers, reducing time-to-first-feedback by 90% while achieving 80% engineer approval. The system uses multiple large language models (from Anthropic, OpenAI, and Google) integrated through a Java-based framework called Aviator, allowing review agents to run within existing services rather than isolated containers and retrieve repository context to improve feedback relevance.

To address quality concerns with early verbose comments, HubSpot implemented a "judge agent" evaluation loop that filters low-value feedback before posting, significantly improving the signal-to-noise ratio. This two-agent approach proved so effective that engineers began requesting Sidekick's feedback before opening pull requests, demonstrating strong adoption and trust in the system. Future improvements include adding persistent memory for review agents and expanding context retrieval across repositories.

### QCon London 2026: SBOMs Move From Best Practice to Legal Obligation as CRA Enforcement Looms - (InfoQ)

https://www.infoq.com/news/2026/03/sbom-viktor-petersson/

# Summary

At QCon London 2026, Viktor Petersson highlighted that Software Bills of Materials (SBOMs) are transitioning from industry best practice to legal requirement due to incoming regulations. The EU's Cyber Resilience Act (enforced starting September 2026) will require SBOMs for any internet-connected products, with full compliance by December 2027, and can block non-compliant products from European markets. Similar mandates are already in place through US Executive Order 14028 for federal software procurement, FDA requirements for medical devices, and PCI-DSS 4.0 for payment card companies.

Petersson emphasized practical SBOM implementation across two main standards—SPDX and CycloneDX—which serve different layers of the software stack. He stressed that high-quality SBOMs require a disciplined four-stage process (creation, augmentation, enrichment, and signing) using domain-specific tools rather than generic scanners, and critically, digital signing in CI/CD pipelines to establish chain of custody. He also noted that most organizations struggle with SBOM lifecycle management, as products often require multiple versioned SBOMs across different components—a challenge comparable to pre-version-control software development practices.

### Presentation: Mobile Server-Driven UI at Scale - (InfoQ)

https://www.infoq.com/presentations/mobile-server-driven-ui-scale/

# Summary

Rafael Ring from Nubank describes how the company implemented server-driven UI at scale to address challenges in their mobile development lifecycle. Nubank, serving over 115 million customers across Latin America with 40+ million daily app users, faced significant pain points with traditional mobile development: managing a single compiled binary across 3,000+ engineers caused dependency conflicts and integration bottlenecks, while the review process through app stores introduced 4-5 week delays before features reached users. To solve this, Nubank evolved from static mobile binaries to a sophisticated framework called Catalyst that uses a tree-walk interpreter in Flutter to dynamically render layouts and logic from JSON payloads sent from the server.

This server-driven UI approach dramatically reduces release cycles and enables rapid feature deployment without requiring app store reviews for every change. By decoupling UI rendering from the binary build process and allowing backend teams to control application behavior through server-side configuration, Nubank achieved greater agility, reduced merge conflicts, and faster time-to-market for features across their massive user base and complex multi-device ecosystem.

### Java 26 Delivers Language Innovation, Library Improvements, Performance and Security - (InfoQ)

https://www.infoq.com/news/2026/03/java26-released/

# Summary

Java 26 has been released with 10 JEPs (Java Enhancement Proposals) focused on language innovation, library improvements, performance, and security. Key features include HTTP/3 support for the HTTP Client API, the eleventh incubation of the Vector API for optimized vector computations, Lazy Constants for flexible immutable value initialization, and Structured Concurrency in its sixth preview. The release also removes the long-deprecated Applet API and introduces Ahead-of-Time Object Caching enhancements under Project Leyden to improve startup and warmup times with any garbage collector.

Five of the ten JEPs remain in preview or incubator stages, indicating ongoing refinement before finalization. Notable security and cryptographic improvements include PEM Encodings for cryptographic objects in its second preview. With Java 27 scheduled for September 2026, the roadmap continues to emphasize developer productivity, performance optimization, and modern programming paradigms through features like structured concurrency and vector computation capabilities.

### QCon London 2026: Use<’lifetimes> For<’what> - (InfoQ)

https://www.infoq.com/news/2026/03/rust-lifetimes-loans-polonius/

# Summary

At QCon London 2026, Ethan Brierley from TrueLayer presented a novel approach to understanding Rust's lifetimes by reframing them through the Polonius borrow checker model. Rather than thinking of lifetimes as code scopes, Brierley proposed viewing them as sets of "loans"—tracking the possible origins of references and their associated terms. This mental model shift makes lifetime-related concepts like subtyping, variance, and the longest function pattern more intuitive and easier to reason about.

Brierley demonstrated how the loans perspective clarifies common borrow checker errors and then built upward to explain more complex topics including lifetime parameters, variance rules for shared and mutable references, and higher-ranked lifetimes using practical examples like the serde library. He concluded by noting that Polonius, still under development, will eventually expand the compiler's acceptance of memory-safe programs currently rejected, while maintaining safety guarantees.

### Article: Beyond Memory Safety: What Makes Rust Different – Lessons from Autonomous Robotics - (InfoQ)

https://www.infoq.com/articles/practical-robustness-going-beyond-memory-safety-rust/

# Summary

This article explores Rust's advantages beyond its well-known memory safety features through the lens of autonomous robotics projects. While memory safety is valuable, Rust's true power lies in its type system and ownership rules, which make it substantially easier to write correct software from the start. The language's minimalist design—eliminating garbage collection, null pointers, inheritance, and other complexity sources—provides a simple foundation centered on data types and functions.

Key features that set Rust apart include its enums with associated data and exhaustive pattern matching, which eliminate null pointer errors and forgotten case handling; the typestate pattern that encodes runtime protocols into types to catch violations at compile time; and ownership/borrowing rules that provide automatic resource management and prevent concurrency bugs. Together, these mechanisms make common developer mistakes difficult to introduce, resulting in inherently more failure-proof code that developers increasingly appreciate once they move beyond the initial learning curve.

### War in Iran Damages Multiple AWS Data Centers, Challenging Multi-AZ Assumptions - (InfoQ)

https://www.infoq.com/news/2026/03/aws-multiaz-conflict-outage/

# Summary

Recent Iranian drone strikes damaged multiple AWS data centers across the Middle East in early March 2026, affecting facilities in the UAE and Bahrain. While AWS's multi-Availability Zone (AZ) architecture is designed to protect against natural disasters like earthquakes and tornadoes by spreading infrastructure across geographically separated locations, this incident revealed that the model had never been tested against military conflict. Two of three AZs in the UAE region and one facility in Bahrain were significantly impaired, disrupting services that relied on redundancy across multiple zones.

The incident prompted significant discussion within the cloud architecture community about fundamental assumptions in disaster recovery planning. Experts highlighted that multi-AZ deployments do not constitute true disaster recovery and that the actual mitigation strategy requires reducing regional exposure rather than switching cloud providers—since geopolitical risks affect all hyperscalers equally in a given region. AWS recommended customers migrate workloads to alternate regions outside the Middle East, though this creates additional challenges around data residency compliance for regulated industries.

### QCon London 2026: Uncorking Queueing Bottlenecks with OpenTelemetry - (InfoQ)

https://www.infoq.com/news/2026/03/queue-otel-observability/

# Summary

Engineers at Gearset struggled to diagnose a delayed backup job despite having comprehensive dashboards, metrics, and logs. They discovered that distributed tracing with OpenTelemetry filled critical visibility gaps by providing hierarchical context across service boundaries. By implementing OpenTelemetry's context propagation for queue clients and attaching trace IDs as message metadata, they could track request journeys end-to-end and identify root causes that metrics and logs alone couldn't reveal.

The team's solution centered on shifting from infrastructure-focused alerts to Service Level Objectives (SLOs) based on customer experience, moving away from misleading metrics like queue size toward latency-based alerting. Using a three-step framework (define SLI, set SLO, configure alerts) and enriching spans with metadata, they enabled discovery-based debugging that uncovered hidden bottlenecks. However, the speakers emphasized that cultural change—convincing teams to adopt tracing by demonstrating real-world value—was ultimately more challenging than the technical implementation.

https://www.infoq.com/news/2026/03/ontology-at-netflix/

They introduced the MELT Layer (Metrics, Events, Logs, Traces) as a unified observability framework and addressed traditional challenges like siloed data sources, disconnected alerting, and difficult troubleshooting—exemplified by a recent incident requiring 30+ engineers from 9 teams and 4 hours to resolve.

### QCon London 2026: From DVDs to Global Streaming How Netflix’s Commerce Architecture Actually Evolved - (InfoQ)

https://www.infoq.com/news/2026/03/netflix-dvds-to-global-streaming/

# Summary

At QCon London 2026, Netflix principal engineer Kasia Trapszo presented how the company's commerce architecture evolved from supporting a US-based DVD rental service to a global streaming platform serving 130+ countries. Rather than a single redesign, Netflix made pragmatic, incremental changes driven by real-world challenges: Brazil's lack of online-enabled debit cards forced a shift from real-time to hybrid payment processing; the 2016 global expansion required rapid but imperfect localization strategies that were refined based on production learnings; and India's regulatory restrictions on storing credit card data necessitated new authentication methods. Each change revealed that early architectural assumptions—predictable payments, single-account billing, distributed signups—no longer held.

As complexity grew, Netflix deliberately scaled its organization with clear domain boundaries and chose to duplicate vendor integration code across teams rather than create shared components, recognizing that "some duplication is cheaper than coordination." The platform's latest evolution came with live events in 2024, which created massive signup spikes requiring graceful degradation strategies that prioritized customer access over immediate financial reconciliation. Trapszo concluded that "great systems don't survive because they were perfectly designed. They survive because they keep evolving as reality changes."

### QCon London 2026: Reliable Retrieval for Production AI Systems - (InfoQ)

# Summary

At QCon London 2026, Lan Chu from Rabobank shared insights from deploying a production AI search system serving over 300 users across 10,000 documents. She emphasized that most RAG (Retrieval-Augmented Generation) system failures stem from indexing and retrieval challenges rather than the language model itself. The system's architecture includes document ingestion with parsing and embedding, retrieval and generation stages, and comprehensive observability monitoring.

Chu highlighted several critical production lessons: accurate document parsing is essential since enterprise documents contain complex layouts (tables, infographics) that plain text conversion misses—requiring visual-language models to preserve structure; chunking strategies must be tested on real data rather than using one-size-fits-all approaches; and retrieval should incorporate signals beyond simple vector similarity, such as temporal scoring and intelligent routing to external APIs. She stressed that robust evaluation frameworks built from real user queries are vital for identifying failure modes and ensuring reliable performance in production systems.

### QCon London 2026: Managing Asynchronous APIs at Scale - (InfoQ)

https://www.infoq.com/news/2026/03/managing-async-apis/

# Summary

At QCon London 2026, Ian Cooper from Just Eat Takeaway presented strategies for managing asynchronous APIs at scale. He highlighted how event-driven architectures start informally but become fragile as organizations grow, making it difficult to discover which services publish and consume events. Cooper proposed addressing these challenges through three pillars: discovery (finding async APIs), governance (ensuring consistent schemas), and provisioning (deploying from specifications).

The solution centers on explicit specifications using technologies like AsyncAPI and xRegistry, combined with standardized metadata formats such as CloudEvents and schema registries for validation. Cooper emphasized that successful implementation requires both tooling and cultural change, enabling teams to generate code, register schemas, and provision infrastructure automatically from endpoint specifications. Tools like EventCatalog make event-driven systems visible and governable, reducing manual coordination between teams and allowing architects better control over production environments—treating async APIs with the same rigor as synchronous ones.

### What Is the PARK Stack? - (Radar)

https://www.oreilly.com/radar/what-is-the-park-stack/

# Summary

The PARK Stack is an emerging open-source software architecture designed specifically for generative AI applications, analogous to how the LAMP Stack served web development in the 1990s. It consists of four core components: PyTorch (for model training and inference), AI models and agents (the foundation of generative AI capabilities), Ray (for distributed programming), and Kubernetes (for cluster management). This stack was first proposed by Ben Lorica as a response to the explosion of interest in large language models and vision models.

The article contextualizes the PARK Stack by comparing it to the LAMP Stack's transformative role in enterprise adoption of open-source software. Just as LAMP became dominant due to flexibility, cost efficiency, and avoiding vendor lock-in, the PARK Stack addresses the specific infrastructure requirements for building and deploying generative AI applications at scale, with PyTorch emerging as the leading framework for training cutting-edge AI models.

### Stop Closing the Door. Fix the House. - (Radar)

https://www.oreilly.com/radar/stop-closing-the-door-fix-the-house/

# Summary

Open source maintainers are increasingly frustrated with low-quality AI-generated pull requests and are closing their doors to external contributors entirely. However, the author argues this is the wrong approach, as AI-assisted coding is now mainstream and maintainers need to adapt rather than exclude contributors.

Instead of rejecting AI PRs, maintainers should prepare their projects by creating clear documentation: a HOWTOAI.md file educating contributors on responsible AI use (what AI should and shouldn't do, accountability expectations, and validation requirements) and an AGENTS.md file with project conventions so AI coding assistants understand the codebase. By improving documentation and setting explicit expectations for both human contributors and AI tools, projects can harness AI's benefits while maintaining code quality.

### Reinventing Python's AsyncIO - (Python Weekly)

https://blog.baro.dev/p/reinventing-pythons-asyncio

# Summary

The author critiques Python's AsyncIO for its unnecessary complexity (unclear primitives like protocols and transports, GIL-related design quirks) and presents TonIO, a new async runtime designed around simplicity. TonIO's core design centers on two basic concepts: **Events** (atomic flags representing what to wait for) and **Waiters** (suspension points that link coroutines to events), eliminating AsyncIO's confusing layers of futures, tasks, and handles.

Leveraging free-threaded Python, TonIO abandons the traditional "main thread principle" entirely—instead of special thread handling, all user code runs on identical worker threads managed by a thread pool, while the main thread handles only low-level I/O. This approach, implemented in Rust for safety and efficiency, provides a Tokio-like experience where developers simply decorate their main function and let the runtime handle concurrency transparently, without needing to understand or manage threads, processes, or AsyncIO's numerous sharp corners.

When migrating from an expensive proprietary model (o3) to a cheaper open-weight model (gpt-oss-120b) for scalability, manual prompt tuning proved inefficient, so they used DSPy's GEPA optimizer to automatically adapt the judge by analyzing failure patterns and iteratively improving prompts.

By combining clear metrics, structured feedback on where the model disagreed with humans, and guardrails against overfitting, Dropbox transformed relevance judging from a fragile manual process into a repeatable, reliable optimization loop that scales across different models.

### django-security-label: A third-party package to anonymize data in your models - (Python Weekly)

https://www.better-simple.com/django/2026/03/18/django-security-label/

# Summary

**django-security-label** is a Django third-party package that leverages PostgreSQL's Anonymizer extension to dynamically mask sensitive data at the database level. Developers define anonymization rules directly on their models using security labels, which automatically mask data based on user roles and scenarios without requiring application-level anonymization logic. The package provides pre-built middlewares like MaskedReadsMiddleware and GroupMaskingMiddleware to control when masking is applied, allowing different user groups (e.g., dispatchers, developers) to see appropriately anonymized versions of the same data.

The package implements masking by temporarily changing PostgreSQL database roles via `SET SESSION ROLE` commands, ensuring that masked data is enforced at the database level rather than in application memory. This "shift-left" security approach allows developers to consider data sensitivity implications during model design. While still in early stages and not yet used in production, the package supports extensive customization through custom middleware implementations, making it adaptable to varied business-specific data anonymization requirements.

### Python Type Checker Comparison: Typing Spec Conformance - (Python Weekly)

https://pyrefly.org/blog/typing-conformance-comparison/

# Summary

Python's type system has evolved across multiple PEPs and different type checkers (mypy, Pyright, Pytype, Pyre), creating inconsistencies in how typing rules are implemented. To standardize this, the Python typing community developed a formal typing specification with a conformance test suite containing ~100 test files. As of March 2026, Pyright leads conformance at 97.8% pass rate, followed by Zuban (96.4%), Pyrefly (87.8%), mypy (58.3%), and ty (53.2%), with Pyright having the fewest false positives and negatives.

However, conformance is just one metric for evaluating type checkers and doesn't capture everything that matters in practice. Gaps in conformance force developers to work around limitations with redundant casts or code restructuring. More importantly, conformance testing doesn't measure critical aspects like type inference quality, performance, IDE integration, error message clarity, and specialized support for third-party packages like Django and Pydantic—all of which significantly impact real-world developer experience.

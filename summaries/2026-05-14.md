### JEP 533 Tightens Exception Handling in Java's Structured Concurrency for JDK 27 - (InfoQ)

https://www.infoq.com/news/2026/05/jep-533-jdk-27/

# Summary

JEP 533 advances Java's Structured Concurrency to integrated status for JDK 27, focusing on improving exception handling and type safety. The key changes include: adopting `ExecutionException` instead of the preview-specific `FailedException` for the standard joiners (aligning with familiar `Future` semantics), adding a third type parameter to `StructuredTaskScope` and `Joiner` to represent exception types thrown by `join()`, and introducing a new `open()` overload that combines the default join policy with configuration operators for cleaner syntax.

These refinements are ergonomic and typing improvements rather than structural redesigns—the core API shape established in Preview 5 remains stable. The changes reduce the conceptual gap between classic concurrency code and structured scopes, make exception contracts more explicit for custom joiners, and simplify configuration for library authors. Teams using earlier previews will need to update exception handling from `FailedException` to `ExecutionException` when migrating to JDK 27.

### Article: The Mathematics of Backlogs: Capacity Planning for Queue Recovery - (InfoQ)

https://www.infoq.com/articles/capacity-planning-queue-recovery/

# Summary

This article explains how to mathematically calculate backlog recovery time in distributed systems using practical formulas rather than guesswork. The key insight is that backlog drain time depends on "surplus capacity" (total processing rate minus arrival rate)—systems provisioned exactly for steady-state traffic have zero recovery capacity and cannot drain backlogs. The article introduces critical concepts like Little's Law (queue_depth = arrival_rate × time_in_queue) and explains why the relationship between utilization and queue growth is non-linear: a 10% traffic spike has minimal impact at 80% utilization but can be catastrophic at 90%.

The article provides practical guidance for capacity planning, including how to identify failure modes like retry amplification and cascading bottlenecks in multi-stage pipelines. It emphasizes that headroom must be calculated using a formula (consumers needed = steady-state consumers + backlog / (processing rate × RTO)) rather than negotiated, and warns that monitoring queue depth across all pipeline stages is essential since backlogs at one stage cascade everywhere.

### Copy Fail and Dirty Frag: Linux Page-Cache Exploits Target Every Major Distribution - (InfoQ)

https://www.infoq.com/news/2026/05/copy-fail-dirty-frag-linux/

# Summary

Two critical Linux kernel vulnerabilities affecting the page cache have been disclosed within a week of each other, targeting all major Linux distributions. Copy Fail (CVE-2026-31431), discovered by Theori using AI-powered security tools, exploits a logic flaw in the algif_aead kernel module that allows unprivileged users to write to the page cache and gain root access. Dirty Frag (CVE-2026-43284 and CVE-2026-43500), disclosed by researcher Hyunwoo Kim, chains two separate vulnerabilities in xfrm-ESP and RxRPC modules to achieve the same outcome. Both vulnerabilities belong to the same bug class as the 2022 Dirty Pipe vulnerability and affect kernels dating back to 2017.

What makes these exploits particularly dangerous is their broad reach and low barrier to entry—they require only an unprivileged local user account with no network access or special privileges, and both kernel APIs are enabled by default in mainstream distributions. The exploits are deterministic (don't rely on race conditions) with high success rates, and proof-of-concept code is minimal. While disclosure timelines were followed for Copy Fail, Dirty Frag's disclosure was expedited due to an embargo breach, forcing maintainers to ask Kim to publish details before patches were available. Major distributions have begun shipping fixes, but the vulnerabilities affect every major Linux distribution tested.

### Cangjie, a New Open-Source Compiled Language with Native Effect Handlers and Algebraic Data Types - (InfoQ)

https://www.infoq.com/news/2026/05/cangjie-effect-handlers-adt/

# Summary

Cangjie (CJ) is a new open-source compiled language developed by Huawei's Programming Languages Lab that aims to be a general-purpose alternative to Java, Kotlin, and Swift. It compiles to machine code, supports multiple platforms (Linux, macOS, Windows, Android, iOS, HarmonyOS), and features static typing, pattern matching, algebraic data types, and concurrent garbage collection. The language is already taught in 80+ Chinese universities.

The most academically significant feature of Cangjie is its native support for **effect handlers**, which generalize exceptions and simplify dynamic binding. Effect handlers use `perform` and `resume` keywords to allow code to interact with its calling context without terminating execution—enabling use cases like dependency injection, mocking, caching, and context-aware logging. This approach allows libraries to adapt their behavior based on their runtime environment (e.g., logging differently on a desktop versus a mobile device) more elegantly than traditional exception handling.

### Netflix Serves 84% of Query Results from Cache with Interval-Aware Caching in Apache Druid - (InfoQ)

https://www.infoq.com/news/2026/05/netflix-druid-interval-cache/

# Summary

Netflix has implemented an interval-aware caching strategy in Apache Druid that achieves an 84% cache hit rate for analytics queries while reducing query load by 33%. The optimization addresses a key inefficiency in rolling window dashboards, where near-identical queries with slightly different time ranges are traditionally treated as separate requests, causing low cache reuse and redundant computation across trillions of rows.

The solution works by decomposing query results into time-aligned segments rather than caching full outputs. When new queries arrive, cached segments for historical data are reused while only the most recent interval is recomputed from Druid and merged with cached results. This approach delivers substantial improvements including 66% faster P90 query times and up to 14x reduction in result bytes, with the system currently deployed as an experimental layer at Netflix with plans for tighter integration into Apache Druid itself.

### From Capabilities to Responsibilities - (Radar)

https://www.oreilly.com/radar/from-capabilities-to-responsibilities/

# Summary

This O'Reilly article argues that high-stakes AI systems—those capable of mutating external state through financial transactions, infrastructure changes, or critical record modifications—must be redesigned around **responsibilities rather than capabilities**. The author critiques the current industry standard of Human-in-the-Loop (HITL) governance, which creates an "Scalability Trap" where approval queues lead to alert fatigue and degraded decision-making as humans rubber-stamp increasing numbers of agent requests.

The proposed solution is **Governance by Exception**, where humans establish policy frameworks that the runtime enforces automatically, with only genuinely exceptional cases escalated for human review. Rather than defining what agents *can* do through capabilities, systems should define what agents *are authorized* to do through stable, assigned responsibilities—similar to role-based access control in traditional software. This shift addresses a critical gap in enterprise AI infrastructure: while the industry has invested heavily in agent capabilities, it has neglected the governance protocols needed for accountability and constraint at scale.

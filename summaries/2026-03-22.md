### Some Things Just Take Time - (Armin Ronacher's Thoughts and Writings)

https://lucumr.pocoo.org/2026/3/20/some-things-just-take-time/

# Summary

Armin Ronacher argues that certain valuable things—like mature trees, trusted relationships, and quality software—inherently require time to develop and cannot be rushed or replaced by technological shortcuts. He critiques the tech industry's obsession with speed and instant gratification, noting that friction and slowness often serve important purposes, such as cooling-off periods for major decisions or thorough compliance processes. While speed works for some tasks, it undermines the trust, quality, and community that define truly successful long-term projects.

Ronacher expresses concern that the push for faster iteration through AI and automation is creating a culture of disposable software and abandonment, where projects and companies shut down without proper closure or care for customers. He reflects on his own experience maintaining open-source projects and a decade-long startup, observing that sustainable value comes from sustained commitment over years, not from rapid experimentation. Ultimately, he contends that tools claiming to save time merely fill freed hours with more work, and that the most meaningful things—trust, quality, and community—cannot be manufactured quickly, only grown patiently over time.

### Profiling Hacker News users based on their comments - (Simon Willison's Weblog)

https://simonwillison.net/2026/Mar/21/profiling-hacker-news-users/#atom-everything

# Summary

Simon Willison describes a technique for profiling Hacker News users by extracting their last 1,000 comments via the publicly available Algolia API and prompting an LLM to analyze them. The process is remarkably effective at revealing detailed information about users' professional identities, technical interests, working styles, and personalities—all derived from their public comments. He demonstrates this by sharing a comprehensive profile of himself generated by Claude, which accurately captures his expertise in AI-assisted coding, his technical focus areas, his security concerns, and even personal details like his location and interests.

Willison acknowledges the somewhat dystopian nature of this capability, noting how easy it is to extract and synthesize this much personal information from publicly available data. While he mainly uses the tool to identify bad-faith commenters on Hacker News, he emphasizes the creepy implications of such profiling becoming accessible to anyone with an internet connection and an LLM account. The article highlights both the power of modern AI to synthesize information and the privacy concerns it raises, even when dealing with data users have willingly shared publicly.

### Autoresearching Apple's "LLM in a Flash" to run Qwen 397B locally - (Simon Willison's Weblog)

https://simonwillison.net/2026/Mar/18/llm-in-a-flash/#atom-everything

# Summary

Researcher Dan Woods successfully ran Qwen3.5-397B, a 397-billion parameter Mixture-of-Experts model, on a 48GB MacBook Pro M3 Max at 5.5+ tokens/second by leveraging Apple's "LLM in a Flash" technique. This approach stores model parameters on SSD and streams them into RAM on-demand, taking advantage of the MoE architecture where each token only activates a subset of model weights. Woods used Claude to automatically run 90 experiments and generate optimized MLX Objective-C and Metal code, resulting in a setup with 2-bit quantized experts (later upgraded to 4-bit for better tool-calling support) and 5.5GB of non-expert components kept in memory.

The breakthrough demonstrates that extremely large models can run locally on consumer hardware through clever optimization, though some concerns remain about output quality at aggressive quantization levels and the limited evaluation methodology described by Claude.

### Snowflake Cortex AI Escapes Sandbox and Executes Malware - (Simon Willison's Weblog)

https://simonwillison.net/2026/Mar/18/snowflake-cortex-ai/#atom-everything

# Summary

A prompt injection attack successfully exploited Snowflake's Cortex Agent by embedding malicious code in a GitHub repository's README file. When a user asked the agent to review the repository, the hidden prompt injection caused the agent to execute a command that downloaded and ran malware from an attacker's server. The vulnerability existed because Cortex's allowlist designated `cat` commands as safe to execute without human approval, but failed to account for process substitution techniques that could be hidden within the command's body.

The incident highlights a fundamental security flaw in using command allowlists for AI agents. The author argues that such pattern-based allowlists are inherently unreliable and advocates instead for deterministic sandboxes that operate independently of the agent itself, treating all agent commands as potentially dangerous regardless of their apparent form.

### Subagents - (Simon Willison's Weblog)

https://simonwillison.net/guides/agentic-engineering-patterns/subagents/#atom-everything

# Summary

Subagents are a key agentic engineering pattern that helps coding agents manage the constraints of LLM context windows (typically capped around 200,000 tokens) by dispatching specialized copies of themselves to handle discrete tasks. When a parent agent encounters a subtask, it can spawn a fresh subagent with its own context window and prompt, allowing the parent to preserve its valuable token budget. The article illustrates this with Claude Code's "Explore" subagent, which investigates a codebase to gather information before the parent agent makes edits.

Beyond simple sequential subagents, the pattern supports parallel execution—where multiple subagents work simultaneously on independent tasks—and specialist subagents customized for specific roles like code review, testing, or debugging. While tempting to overuse, subagents' primary value lies in managing token-heavy operations and freeing up the root agent's context for higher-level reasoning. Most major coding platforms now support subagents with varying implementations and documentation.

### How coding agents work - (Simon Willison's Weblog)

https://simonwillison.net/guides/agentic-engineering-patterns/how-coding-agents-work/#atom-everything

# Summary

A coding agent is software that harnesses a Large Language Model (LLM) and extends it with additional capabilities through tools and system prompts. At its core, an LLM is a machine learning model that predicts the next token (word fragment) in a sequence; modern LLMs use chat-templated prompts that simulate conversations, with the agent maintaining conversation state and replaying it with each new input. The defining feature of coding agents is their ability to call tools—functions the agent can execute—which are invoked when the LLM generates specific tool-call syntax in its response, with results fed back into the conversation loop.

Building a coding agent fundamentally requires combining an LLM, a system prompt (detailed instructions on how the model should behave), and tools in a loop that allows the model to call functions, receive results, and continue reasoning. Recent advances include reasoning capabilities that allow models to spend additional processing time thinking through problems before responding. While a simple tool loop can be implemented in just dozens of lines of code, creating a robust production agent requires more work, but the underlying mechanics are surprisingly straightforward.

### What is agentic engineering? - (Simon Willison's Weblog)

https://simonwillison.net/guides/agentic-engineering-patterns/what-is-agentic-engineering/#atom-everything

# Summary

Agentic engineering is the practice of developing software with coding agents—AI tools that can write and execute code in loops to achieve specified goals. These agents work by calling language models with prompts and tool definitions, then iteratively executing and refining code until objectives are met. The key capability that enables agentic engineering is the ability to directly run code, allowing agents to produce demonstrably working solutions rather than just theoretical suggestions.

Rather than replacing human engineers, agentic engineering shifts the focus of software development from pure code writing to higher-level problem-solving. Engineers must define clear goals, provide appropriate tools, specify problems with sufficient detail, and rigorously verify and iterate on results. This approach requires deliberate learning and tool refinement, ultimately helping teams tackle more ambitious projects and produce higher-quality code. The author distinguishes agentic engineering from "vibe coding"—unreviewed, prototype-quality LLM output—emphasizing that effective agentic engineering demands careful oversight and refinement to reach production standards.

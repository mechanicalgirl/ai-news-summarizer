### How LinkedIn Identified a Kernel Lock Contention Issue Causing Recurring System Freezes - (InfoQ)

https://www.infoq.com/news/2026/05/linkedin-kernel-lock-freeze/

# Summary

LinkedIn engineers encountered recurring 10-15 second database outages with no clear external triggers or useful logs. Traditional monitoring tools failed to identify the root cause, so they developed an innovative solution using eBPF off-CPU profiling that automatically captured kernel stack traces the moment a freeze was detected, revealing that a massive 3.5 GB memory allocation was triggering a kernel-level lock on the mmap_lock semaphore, blocking all threads from memory operations.

The issue stemmed from a Rust HashMap (pkey_vs_docref) that doubled in size when reaching 58 million entries, causing the sudden memory spike. LinkedIn resolved it by pre-allocating the HashMap to prevent runtime resizing, accepting an additional 3 GB memory cost at startup. The incident demonstrated that eBPF-based profiling and automated instrumentation triggered on failure conditions are essential for diagnosing ephemeral, silent system freezes.

### Sarang Kulkarni on Lessons from Building Deep Research Agents in Production - (InfoQ)

https://www.infoq.com/news/2026/05/kulkarni-deep-research-agents/

Sarang Kulkarni from Thoughtworks presented lessons learned from building Deep Research Agents—AI systems designed to conduct multi-step research for complex tasks in critical industries like healthcare and pharmaceuticals. He highlighted that traditional Q&A models are insufficient; researchers need systems that can discover, connect, and reason across internal and internet data while maintaining reliability and compliance. The team evolved from a simple RAG chatbot to an "Agentic RAG++" solution with three core loops: clarification, research (with think-plan-execute-reflect-adjust cycles), and writing loops.

### Pullfrog AI: Open-Source CodeRabbit Alternative Powered by GitHub Actions - (InfoQ)

https://www.infoq.com/news/2026/05/pullfrog-ai-github/

# Summary

Pullfrog AI is an open-source GitHub bot created by Colin McDonnell (author of Zod) that serves as a model-agnostic alternative to CodeRabbit for AI-powered code review and development automation. Running entirely within GitHub Actions, Pullfrog uses a bring-your-own-key approach, allowing developers to connect any LLM provider (Anthropic, OpenAI, Google, Mistral, DeepSeek, etc.) and switch between models via configuration. The tool can be triggered manually by tagging @pullfrog or automated through configurable events like pull requests, CI failures, and review submissions.

Beyond code review, Pullfrog differentiates itself by offering broader functionality including issue triage, CI remediation, merge conflict resolution, and plan generation. It includes built-in capabilities for git/GitHub operations and a headless browser tool for end-to-end testing. The project has gained significant community interest with over 1,000 likes since its announcement, and McDonnell plans to add a CLI for local development workflows, though Pullfrog is primarily designed as a CI-based agent rather than a local development tool.

https://www.infoq.com/articles/architecting-cloud-native-kafka/

# Summary

This article discusses how Apache Kafka is evolving from a traditional hardware-bound system to a cloud-native, disaggregated architecture driven by cloud economics. The shift from fixed infrastructure costs to per-request cloud API charges fundamentally changes Kafka's operational model, requiring platform teams to implement telemetry-driven cost tracking and replay governance. Key architectural innovations include tiered storage, improved rebalancing protocols for Kubernetes autoscaling, virtual clusters for better multi-tenancy isolation, and Share Groups that decouple partition count from consumer parallelism.

The article uses Discover Financial Services' migration as a case study, showing how cloud-native Kafka enabled dramatic improvements in operational efficiency (reducing pricing change adoption from six months to three weeks). However, this cloud migration also exposed new financial challenges, particularly around storage replication across availability zones and historical data retention costs. The article emphasizes that successful cloud-native Kafka deployments require architects to treat these systems as "economic operating systems" with active cost visibility, client-level expense attribution, and cost-aware governance policies to prevent unexpected bill spikes.

# Summary

TamboUI, a new modern Terminal User Interface (TUI) library for Java, has launched to address the need for better terminal UI development capabilities in the Java ecosystem. Inspired by Rust's Ratatui library but offering higher-level abstractions, TamboUI provides APIs ranging from low-level terminal drawing to high-level components and event handling, similar to JavaFX for desktop applications. The creators emphasize that Java developers deserve better terminal tooling and that the ecosystem should move beyond viewing Java as only suitable for enterprise/microservice use cases, since Java's own core tools already live in the terminal.

### Article: The Schema Proliferation Problem in Kafka and Flink Pipelines: How to Solve It - (InfoQ)

https://www.infoq.com/articles/schema-proliferation-problem/

# Summary

This article addresses the "schema proliferation problem" that occurs in Kafka and Flink data pipelines when teams use a one-to-one mapping between event types and schemas. As systems scale with dozens of event types, each variant creates separate schemas and downstream tables, leading to complex queries, maintenance overhead, and schema drift. For example, a ride-sharing platform tracking driver activity across four event types and three ride types results in twelve separate schemas and tables instead of a consolidated approach.

The solution presented is **discriminator-based schema consolidation**, which consolidates related event schemas (that share 80-95% structural similarity) into fewer schemas using discriminator enum fields. This approach reduces table count dramatically (from 12+ tables to 2, for example), simplifies consumer queries to single-table operations, and enables backward-compatible schema evolution through nullable attribute blocks. The article recommends designing schemas around consumer access patterns and using layered adapter designs to separate transformation logic from framework integration, making the consolidation pattern easier to implement and maintain in existing Flink pipelines.

### Who Authorized That? The Delegation Problem in Multi-Agent AI - (Radar)

https://www.oreilly.com/radar/who-authorized-that-the-delegation-problem-in-multi-agent-ai/

# Summary

The article addresses a critical security gap in multi-agent AI systems: the "delegation problem." As enterprises deploy interconnected AI agents that call other agents through protocols like MCP (Model Context Protocol) and A2A, they can track connectivity but lack proper authorization frameworks for delegation chains. When Agent A calls Agent B, which calls Agent C, current systems cannot definitively verify whether each downstream agent was actually authorized to perform its specific actions—creating a silent security leak rather than a loud failure.

The core issue is that existing authorization models (OAuth 2.1, static API keys, service accounts) were designed for single-client scenarios or simple service-to-service communication, not for complex agent delegation hierarchies. These tools don't address whether a specific downstream action was legitimately derived from an upstream instruction, leaving enterprises unable to answer fundamental security questions like "who authorized this agent to access that data?" As agent ecosystems evolve faster than authorization frameworks, organizations must develop delegation-aware security models to govern multi-agent workflows properly.

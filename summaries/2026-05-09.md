### Behavior-Oriented Concurrency in Python - (Python Weekly)

https://microsoft.github.io/bocpy/

# Summary

Behavior-Oriented Concurrency (BOC) is a new programming paradigm for Python that eliminates the need for locks and deadlocks by giving each behavior (decorated function) unique temporal ownership of shared data. Instead of managing complex lock coordination, programmers focus on organizing data flow through functions, resulting in simpler, more maintainable code that still achieves true multi-core parallelism through efficient scheduling across multiple processes.

BOC is built on two key concepts: cowns (concurrent-owned variables) that ensure only one interpreter accesses data at a time using lock-free atomic operations, and behaviors that are automatically scheduled to run once their required cowns are available. The article demonstrates this through a cooking example, showing how the BOC approach produces cleaner, more intuitive code compared to traditional thread-based solutions that require explicit lock management and condition variables.

### Fast Mesh Booleans in Python - (Python Weekly)

https://polydera.com/tutorials/fast-mesh-booleans-in-python

# Summary

This tutorial introduces **trueform**, a Python library for performing fast mesh boolean operations (union, intersection, and difference) on large polygon meshes at interactive speeds. With a single pip install, users can load meshes as NumPy arrays, apply transformations, and execute boolean operations in milliseconds—for example, completing three boolean operations on 500K-polygon meshes in under 14ms each. The library handles real-world, non-ideal meshes without requiring preprocessing.

The tutorial demonstrates practical workflows including loading mesh files, creating transformed copies using shared views, running boolean operations, and extracting results with face labels for attribute tracking. For efficiency with multiple operations on the same geometry, users can precompute spatial and topological structures once and reuse them across transformed views, enabling 10 booleans on million-polygon meshes in under 140ms. Additional features include extracting intersection curves and robust handling of common mesh imperfections like non-manifold edges and inconsistent winding.

### Django security releases issued: 6.0.5 and 5.2.14 - (Python Weekly)

https://www.djangoproject.com/weblog/2026/may/05/security-releases/

# Summary

The Django team has released security updates for versions 6.0.5 and 5.2.14 to address three low-severity vulnerabilities. The vulnerabilities include: a denial-of-service risk in ASGI requests where missing Content-Length headers could bypass file upload limits, a session fixation vulnerability in cached public pages when SESSION_SAVE_EVERY_REQUEST is enabled, and improper caching of requests with "Vary: *" headers that could expose private data. All users are encouraged to upgrade immediately.

Patches have been applied to the main, 6.0, and 5.2 branches, with detailed changesets available for each CVE. The Django team reminds users that security issues should be reported privately to security@djangoproject.com rather than through public channels.

### Boosting multimodal inference performance by &gt;10% with a single Python dictionary - (Python Weekly)

https://modal.com/blog/boosting-multimodal-inference-performance-by-greater-than-10-with-a-single-python-dictionary

# Summary

The authors identified and fixed a performance bottleneck in SGLang's multimodal inference scheduler by profiling it with py-spy. They discovered that the scheduler was repeatedly calling `torch.UntypedStorage._new_shared_cuda` to access GPU memory pools across processes, incurring expensive host-side overhead on book-keeping operations that could be done just once. The solution was elegantly simple: replace the repeated calls with a Python dictionary cache that stores pool handles, avoiding the need to reopen the same handles multiple times per scheduler iteration.

The optimization delivered significant real-world improvements: 16% higher throughput (22.2 to 25.7 req/s), 13% lower mean time-to-first-token, and 10% lower end-to-end latency when benchmarking Qwen2.5-VL-3B-Instruct on an H100 GPU. The fix exemplifies the principle of inference optimization—never block the GPU by eliminating unnecessary host-side overhead—and is now available in SGLang v0.5.10.

### Handling Schema Issues in Polars - (Python Weekly)

https://pola.rs/posts/schema-evolution/

# Summary

This article explains how to handle schema changes in Polars across different data formats. Schema changes come in four types: additive (new columns), subtractive (missing columns), type drift (dtype changes), and breaking changes (incompatible transformations). The article provides specific parameters and strategies for each format—CSV uses `schema_overrides`, `schema`, `infer_schema`, and `ignore_errors` to manage type inference; Parquet uses `extra_columns`, `missing_columns`, and `ScanCastOptions` to handle multi-file schema divergence; and Delta Lake and Apache Iceberg use `schema_mode="merge"` and `update_schema()` respectively.

The key takeaway is that identifying the type of schema change determines which Polars parameter to use. For production pipelines, `schema_overrides` is recommended for CSVs, while `missing_columns="insert"` combined with `ScanCastOptions(integer_cast="upcast")` handles most Parquet issues. For unpredictable file ordering or complex scenarios, `pl.concat(..., how="diagonal_relaxed")` automatically coerces types to a common supertype while aligning columns by name.

### Using Bag-of-Words With PyCharm - (Python Weekly)

https://blog.jetbrains.com/pycharm/2026/04/using-bag-of-words-with-pycharm/

# Summary

This article explains the bag-of-words (BoW) model, a foundational natural language processing technique that converts text into numerical vectors by counting word frequencies while ignoring grammar and word order. Despite its simplicity and age, BoW remains effective for many NLP tasks because the presence of certain words often signals meaning more strongly than their arrangement. The article walks through how BoW works—tokenizing text, creating a vocabulary, and encoding documents as vectors of word counts—while highlighting its key advantages: computational efficiency, ease of implementation, and effectiveness for tasks like sentiment analysis and document classification.

The article also advocates for using PyCharm as an ideal IDE for BoW projects, as its code intelligence features, debugging tools, Jupyter Notebook integration, and project management capabilities streamline the iterative workflow of text preprocessing, vectorization, and validation. The piece concludes by announcing a hands-on project that will demonstrate these concepts in action, classifying news articles into four categories using BoW techniques in PyCharm.

### Full-Text Search with DuckDB - (Python Weekly)

https://peterdohertys.website/blog-posts/full-text-search-w-duckdb.html

# Summary

This article explores full-text search (FTS) capabilities in DuckDB, a tool that extends basic SQL queries with more sophisticated search features. The author explains FTS fundamentals like stemming (matching word variations), stop word removal, and Okapi BM25 scoring algorithms, while acknowledging that DuckDB's current FTS implementation, though solid, lacks some advanced features like phrase queries and result highlighting that exist in more mature solutions like Elasticsearch and PostgreSQL.

The article provides a practical walkthrough of implementing FTS on a dataset of 13,000 emails, including Python code for pre-processing raw .eml files into JSON, importing them into DuckDB, creating an FTS index, and executing various queries with different parameters. The author demonstrates how to use BM25 parameters (k₁ for term frequency and b for document length normalization) to fine-tune search results and filter out transactional emails and mailing lists, showing DuckDB as a capable option for full-text search despite its limitations compared to specialized search engines.

### lightning PyPI Compromise: A Bun-Based Credential Stealer in Python - (Python Weekly)

https://snyk.io/blog/lightning-pypi-compromise-bun-based-credential-stealer/

# Summary

On April 30, 2026, two malicious versions (2.6.2 and 2.6.3) of the popular Lightning PyPI package were published, compromising a deep learning framework with over 7.9 million monthly downloads. The malicious wheels contained a hidden `_runtime` directory that downloads the Bun JavaScript runtime and executes an ~11 MB obfuscated credential stealer upon import. The stealer harvests GitHub tokens, npm credentials, and cloud metadata while using GitHub's GraphQL API to poison repositories and propagate itself to npm packages. PyPI has since quarantined the project.

This attack represents the second consecutive day of Bun-based stealer incidents targeting major software ecosystems, linked by identical obfuscation techniques and the same Bun version loader. The compromise was likely enabled through a compromised PyPI API token rather than the GitHub Actions workflow itself, as the project lacked OIDC-based Trusted Publisher authentication. Organizations that imported Lightning 2.6.2 or 2.6.3 should treat their environments as exposed to credential theft and potential repository poisoning.

### How we rebuilt search ranking at Faire with deep learning - (Python Weekly)

https://craft.faire.com/how-we-rebuilt-search-ranking-at-faire-with-deep-learning-14f080679c83

# Summary

Faire, an online wholesale marketplace, rebuilt its search ranking system by transitioning from XGBoost tree-based models to deep learning to address growing complexity. The previous system struggled with balancing multiple objectives (relevance, inventory freshness, brand discovery), modeling sequential user intent, handling position bias, and sharing learnings across different search surfaces. Rather than a complete rewrite, the team took an incremental approach, establishing an end-to-end workflow with three components: a training data pipeline, a model training framework, and serving infrastructure.

The rebuild required rethinking foundational aspects across the entire stack. Key challenges included addressing severe data quality issues through enhanced observability and validation checks, and rethinking feature normalization since neural networks are sensitive to feature scaling—unlike tree-based models. The team moved from simple MinMax scaling to distribution-aware normalization techniques (log transforms and Z-score normalization) to preserve signal resolution. This two-year effort unlocked new capabilities including multi-task learning and cross-surface fine-tuning, enabling Faire to better optimize multiple ranking objectives simultaneously and share learnings across different search surfaces.
